# SnapCode · 用户使用指南

[返回项目主页（GitHub README）](../README.md)

SnapCode 是一款常驻系统托盘的 Windows 小工具：它监听「手机连接」（Phone Link）同步到 Windows 通知中心的短信通知，自动提取其中的验证码并保存到本地历史；收到验证码时即刻复制到剪贴板，之后在任意应用中按下 `Ctrl+Shift+V`（可自定义），就能把最新一条验证码直接粘贴到当前输入框——不用再拿起手机，也不用在窗口间来回切换。

## 功能一览

- **自动捕获**：实时监听 Windows 系统通知，「手机连接」推送的短信一到即被识别
- **智能识别**：自动提取常见中英文文案中的验证码（如「验证码 123456」「code is 123456」）
- **收到即复制**：识别到验证码后自动写入剪贴板（可在设置中关闭）
- **一键粘贴**：按下 `Ctrl+Shift+V`，最新验证码直接粘贴到当前焦点输入框
- **历史记录**：本地保存验证码记录，可查看、复制、删除，支持保留 1 / 3 / 7 / 30 天或永久
- **来源可扩展**：默认监听「手机连接」，也可添加其他手机互联应用作为通知来源
- **邮箱验证码**：除短信外，还能添加多个邮箱账户，通过 POP3 轮询或 IMAP（IDLE 实时推送）自动识别新邮件中的验证码
- **安静常驻**：关闭窗口后最小化到系统托盘继续监听，支持暂停监听与开机自启
- **自动更新**：启动时自动检查新版本，更新包经签名校验，一键安装、自动重启，是否更新由你决定
- **深浅主题**：浅色 / 深色 / 跟随系统三种外观

## 环境要求

- Windows 10 1809 或更高版本 / Windows 11
- WebView2 运行时（Windows 11 及较新的 Windows 10 通常已内置；若缺失，请从微软官网下载安装 Evergreen Bootstrapper）

以上为运行要求。Node.js、Rust 等仅为开发依赖，普通用户无需安装；如需从源码自行构建，见 [README.md](../README.md) 的 Getting started 部分。

## 安装

1. 打开 GitHub 仓库的 [Releases 页面](https://github.com/Dream-XJ/snapcode/releases)；
2. 下载最新版本的 Windows 安装包（`.msi` 或 `.exe` 均可）；
3. 双击安装包按提示完成安装，然后从「开始」菜单启动 **SnapCode**。

## 授予通知权限

SnapCode 依赖 Windows 的通知访问权限才能读取短信通知：

1. **首次启动**时系统会弹出通知访问请求窗口，请点击「允许」。
2. 如果错过了弹窗或需要手动开启：
   - 打开 Windows **设置 → 隐私和安全性 → 通知**；
   - 打开「**允许应用访问通知**」（部分版本称「通知访问权限」）；
   - 在应用列表中找到 **SnapCode**，打开其开关。
3. 应用内也提供了入口：权限被拒绝时，设置页「通知监听」区会显示「**打开系统设置**」（直达上述系统设置页）和「**重新检测**」（授权后点击刷新状态）。

权限状态会实时显示在主界面顶部。未授权时应用仍可打开，但无法捕获验证码。

## 手机端前提

SnapCode 本身不与手机直接通信——短信是经由「手机连接」同步进 Windows 通知的，因此需要先在手机与电脑之间完成配对。

### iOS（iPhone）

- 要求 **iOS 16 或更高版本**；
- 在电脑的「**手机连接**」应用中完成与 iPhone 的配对；
- 使用过程中 iPhone 与电脑需保持**蓝牙连接**（iOS 的短信同步依赖蓝牙）；
- iPhone 上进入 **设置 → 通知 → 信息 → 显示预览**，设为「始终」或「解锁时」——若设为「从不」，通知正文中不含短信内容，SnapCode 无从提取验证码。

### Android

- 在手机的「**连接至 Windows**」（Link to Windows）应用中与电脑完成配对，并开启**短信同步**；
- 确认短信通知能够推送到 Windows 通知中心。

## 使用说明

### 首次引导

首次启动会显示引导页，简要介绍「监听通知 → 识别验证码 → 快捷粘贴」的工作方式，并引导你完成通知权限授权。授权完成后点击「**我已完成授权，开始使用**」即可进入主界面。

### 托盘与窗口

- **左键单击托盘图标**：打开主窗口；
- **右键单击托盘图标**：弹出菜单——**打开主窗口** / **暂停监听**（暂停时显示「恢复监听」）/ **退出**；
- **关闭主窗口不会退出程序**，只是隐藏到托盘继续监听；在托盘菜单选择「退出」才会真正结束程序。

### 粘贴验证码

在任意应用的输入框中按下 `Ctrl+Shift+V`：SnapCode 会取出最新一条验证码写入剪贴板，并模拟一次 `Ctrl+V` 将其粘贴到当前焦点输入框。

快捷键可在设置中修改，格式为 `修饰键+键名`（如 `Ctrl+Shift+V`、`Alt+C`）。若与其他程序的快捷键冲突，保存时会提示失败，更换组合后重试即可。

### 历史记录

主界面按时间列出捕获到的验证码，可逐条复制或删除；过期记录会按「保留策略」自动清理，也可在设置中一键清空全部历史。

### 设置项一览

| 设置项 | 说明 | 默认值 |
| --- | --- | --- |
| 主题 | 浅色 / 深色 / 跟随系统 | 跟随系统 |
| 粘贴快捷键 | 粘贴最新验证码到当前焦点 | `Ctrl+Shift+V` |
| 自动复制 | 识别到验证码即写入剪贴板 | 开启 |
| 开机自启 | 登录 Windows 后自动启动 | 开启 |
| 通知来源 | 监听的应用来源（AUMID），可添加多个，包含匹配 | `Microsoft.YourPhone_8wekyb3d8bbwe`（手机连接） |
| 邮箱验证码 | 多账户管理，每账户可选 POP3 轮询或 IMAP（IDLE 实时推送）识别新邮件验证码 | 关闭 |
| 保留策略 | 历史记录保留 1 / 3 / 7 / 30 天或永久保留 | 7 天 |
| 清空历史 | 一键删除全部历史记录 | — |
| 模拟通知 | 输入一段文本，验证解析与入库流程 | — |
| 列出系统通知 | 列出当前系统通知的真实 AUMID，可一键加为来源 | — |

### 自定义来源 AUMID

AUMID（应用用户模型 ID）是 Windows 用来标识通知来源应用的 ID。SnapCode 只处理「通知来源」列表中应用发出的通知，匹配规则为**大小写无关的包含匹配**——例如默认的 `Microsoft.YourPhone_8wekyb3d8bbwe` 可以同时命中 `Microsoft.YourPhone_8wekyb3d8bbwe!App` 这类带后缀的变体。

通过添加其他 AUMID，SnapCode 也能捕获「手机连接」以外的手机互联软件（如厂商自家的互联应用）推送的短信通知：

1. 让目标应用推送一条通知，并让它保留在 Windows 通知中心；
2. 打开 SnapCode **设置 → 调试 → 列出系统通知**，找到该应用的通知，确认其真实 AUMID；
3. 点击通知卡片上的「**加为来源**」（也可以在「通知来源」处手动输入 AUMID 添加）。

此后该应用推送的含验证码短信就会被自动捕获。

> 误删了默认的「手机连接」来源？此时「通知来源」列表底部会出现「**恢复默认来源（手机连接）**」，点击即可一键加回。

### 邮箱验证码

除了短信，SnapCode 还能监听邮箱中的验证码：可同时添加**多个邮箱账户**，每个账户独立选择 **POP3（轮询）** 或 **IMAP（IDLE 实时推送）**，识别新邮件里的验证码，走与短信完全相同的「入库 → 复制 → 快捷键粘贴」流程。

**第一步：开启邮箱的 POP3 / IMAP 服务并获取授权码**

大多数邮箱默认关闭客户端协议，需要先在网页版邮箱设置中开启，并生成「客户端授权码」（不是登录密码）：

- **QQ 邮箱**：设置 → 账户 →「POP3/IMAP/SMTP/Exchange/CardDAV/CalDAV服务」→ 按需开启 POP3/SMTP 服务或 IMAP/SMTP 服务，按提示生成授权码；
- **163 邮箱**：设置 →「POP3/SMTP/IMAP」→ 按需开启 POP3/SMTP 服务或 IMAP/SMTP 服务，按提示生成授权码；
- **Outlook**：一般直接使用账户密码（开启两步验证的账户需使用应用专用密码），服务器为 `outlook.office365.com`；
- **Gmail 暂不支持**：Gmail 的 IMAP/POP3 强制使用 OAuth2 登录，本版本无法连接，请勿添加 Gmail 账户。

**第二步：在 SnapCode 中添加账户**

打开 **设置 → 邮箱验证码**，点击「**添加账户**」，在展开的编辑器中填写：

| 配置项 | 说明 | 示例 |
| --- | --- | --- |
| 备注名 | 账户显示名，可选；留空时显示账号地址。历史记录的来源会带上它，便于区分是哪个账户捕获的 | `工作邮箱` |
| 协议 | `POP3（轮询）` 或 `IMAP（IDLE 实时推送）`，选择建议见下方 | IMAP |
| 服务器 / 端口 | 服务器地址与端口；切换协议时端口在 995 ↔ 993 间自动联动 | `imap.qq.com` / `993` |
| 账号 | 一般是完整邮箱地址 | `you@qq.com` |
| 授权码 | 第一步生成的客户端授权码（非登录密码） | — |
| 轮询间隔 | 多久检查一次新邮件（30 秒 / 1 分钟 / 2 分钟 / 5 分钟）；IMAP 仅在服务器不支持 IDLE 时按此间隔轮询 | 1 分钟 |
| SSL/TLS | 保持开启；仅明文连接时关闭（POP3 端口 110 / IMAP 端口 143） | 开启 |

填完点击「**测试连接**」验证配置是否正确，成功后会显示邮箱中的邮件总数。

保存后账户以卡片列出：卡片上有状态点、显示名、协议徽标（POP3 / IMAP）和启停开关，可随时展开编辑或删除。账户的添加、修改、删除**即时生效，无需重启**；顶栏的「暂停」会同时暂停通知监听与所有邮箱账户。

**POP3 还是 IMAP？**

- **IMAP（推荐）**：服务器支持 IDLE 时新邮件**实时推送**，验证码几乎零延迟；不支持 IDLE 的服务器自动降级为按轮询间隔检查，行为与 POP3 相同；断线后自动重连。
- **POP3（轮询）**：按固定间隔检查新邮件，兼容性最广；时效取决于轮询间隔。

**注意几点：**

- **首次启用某个账户不会导入历史邮件**：该账户第一次连接只建立基线，之后到达的新邮件才会被识别；基线按账户独立，删除账户后重新添加会重新建立基线；
- 每个账户的状态独立显示在卡片状态点上：绿色为**监听中**，灰色为**已暂停 / 未启用**，红色为**轮询出错**（附错误原因）；
- 授权码与其他设置一样**明文保存在本机** `settings.json` 中，不会上传到任何地方（见「隐私说明」）。

### 调试工具

设置页底部的「调试」区提供两个排障工具：

- **模拟通知**：输入一段文本（例如「【微信】您的验证码是 482913，5 分钟内有效。」），走完整的识别与入库流程，用来验证解析效果；
- **列出系统通知**：列出当前 Windows 通知中心的 Toast 及其真实 AUMID，抓不到验证码时可用它确认来源是否配置正确，并可一键「加为来源」。

### 应用更新

SnapCode 会在每次启动时自动检查新版本（离线等检查失败的情况不会打扰你）。发现新版本时会弹出提示框，展示版本号与更新说明：

- 点击「**立即更新**」开始下载更新包，下载完成后安装程序自动接管并完成更新，应用随即重启为新版本；
- 点击「**暂不更新**」关闭提示，下次启动时会再次检查提醒。

也可以随时在 **设置 → 关于 → 版本更新** 处点击「检查更新」手动检查。所有更新包均经过签名校验，且只来自本项目的 GitHub Releases。

## 隐私说明

所有验证码与历史记录**仅保存在本机**（本地 SQLite 数据库），SnapCode 不进行任何网络上传，也不收集任何数据。你可以随时删除单条记录、一键清空历史，或通过保留策略控制数据留存时长。如配置了邮箱监听，各邮箱账户的授权码同样仅保存在本机 `settings.json`（明文），不会发送到对应邮箱服务器之外的任何地方。

## 常见问题

**收不到验证码？**

- 确认「手机连接」中的手机处于**已连接**状态，且短信同步已开启；
- 确认 Windows 通知访问权限已授予 SnapCode（见上文「授予通知权限」）；
- 确认 Windows 通知中心里确实出现了这条短信通知——若通知中心没有，问题出在手机同步一侧；
- 先用设置页调试区的「**模拟通知**」验证识别与入库链路是否正常，再用「**列出系统通知**」确认来源 AUMID 是否在监听列表中。

**邮箱验证码收不到？**

- 先在对应账户的编辑器里点「**测试连接**」验证配置：失败原因会直接提示（连不上服务器 / 授权码错误），QQ/163 等邮箱必须填**授权码**而非登录密码；
- 看账户卡片上的状态点：红色为出错（附错误原因），灰色为已停用或全局暂停中；
- 确认邮件是在**该账户启用之后**到达的——每个账户首次启用都不会导入历史邮件；
- 确认网页版邮箱的 POP3 / IMAP 服务仍处于开启状态，部分邮箱关闭二次验证后授权码会失效，需重新生成；
- 使用 IMAP 时新邮件通常即时到达；若服务器不支持 IDLE 或使用 POP3，时效取决于轮询间隔，间隔不宜小于 30 秒，过于频繁可能被服务器拒绝；
- Gmail 的 IMAP/POP3 强制 OAuth2 登录，本版本不支持，添加 Gmail 账户会一直报错。

**iPhone 收不到验证码，或用着用着突然收不到了？**

- **蓝牙断开是最常见的原因**：iOS 的短信同步依赖蓝牙，请检查 iPhone 与电脑的蓝牙连接，必要时在「手机连接」中重新配对；
- 检查 iPhone 的 **设置 → 通知 → 信息 → 显示预览** 是否被设为「从不」，改为「始终」或「解锁时」。

**快捷键没反应？**

- 快捷键可能被其他程序占用：到设置中更换为其他组合即可（被占用时保存会失败并提示）；
- 确认 SnapCode 正在运行（托盘中有图标），且监听未处于暂停状态。

**按了快捷键，但目标应用里没有粘贴进去？**

- 如果目标应用是**以管理员身份运行**的，Windows 的安全机制会阻止普通权限的 SnapCode 向它注入按键。此时验证码仍会复制到剪贴板，在该输入框中手动按 `Ctrl+V` 即可；
- 若经常需要向这类应用粘贴，可以右键 SnapCode 的快捷方式，选择「以管理员身份运行」。

**为什么有时验证码会晚一两秒才出现？**

- 在部分 Windows 版本上，系统不允许未打包（无包身份）的应用订阅通知实时事件。此时 SnapCode 会自动降级为**每 1 秒轮询一次**系统通知，功能完全等价，通常只晚 1 秒左右，属正常现象，无需处理。

**窗口关了程序还在吗？**

- 在。关闭窗口只是隐藏到系统托盘，监听继续进行；托盘菜单选择「退出」才会真正结束程序。

**检查更新失败？**

- 更新检查需要访问 GitHub Releases。网络直连 GitHub 不稳定时，请开启代理软件的「系统代理」或 TUN 模式后重试；v0.2.0 的更新检查不支持系统代理（始终直连），请升级到 v0.2.1 或更高版本；
- 新版本刚发布后的几分钟内，`releases/latest` 可能仍指向旧版本而返回 404，稍后重试即可；
- 错误信息会附带具体原因（HTTP 状态码或网络错误），可据此排查；也可以直接到 [Releases 页面](https://github.com/Dream-XJ/snapcode/releases) 手动下载安装包覆盖安装。
